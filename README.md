# FORTH80　 ver. 0.5.8

### Abstract

An emulator of the programming language FORTH.

FORTH80 compliants with FORTH-79 standard.

----
## これは何？

FORTH というプログラミング言語の処理系の、一種のエミュレータです。ただし、あえて古い形式のものをエミュレートしています。

FORTH-79 Standard に準拠しています。

これは処理系そのものではなくそのエミュレータであることに注意してください。しかしながら、ターミナルをいわば遠隔操作する機能を付加したことにより、まったく孤立した「箱庭」ではなくなったので、スクリプト言語のように使うことも不可能ではないかもしれません。

井上外志雄『標準 FORTH』(共立出版，1985）で解説されている FORTH 言語の仕様に従っています。それゆえ、この書籍を読む際には少なからず役立つと思います。

## 使い方

以下のファイルを同一フォルダ内に置いてください。専用のフォルダを用意することをお奨めします。

- VSM, VSM.exe のどちらか一方
- FORTH80.bin
- DRIVE0.img
- DRIVE1.img

VSM.exe は Windows 用、VSM は Mac 用です。Mac 版のソースコードを再コンパイルすれば（ソースファイル VSM_mac.c。ただし同梱の conio.h が必要） Linux でも動くと思います。

Windows の場合は、VSM.exe をクリックすればコンソール画面が立ち上がります。Mac の場合は、ターミナルから ./VSM と入力して起動しないとダメです。

それ以前に、「このアプリは安全が保証されていないので実行中止」の趣旨のダイアログ、もしくはメッセージが出るかもしれません。そのときは、Windows の場合は、ダイアログの「詳細」のところをクリックして出てくる「実行」ボタンをクリックしてください。Mac の場合は、Control キーを押しながら VSM をクリックして「開く」を選択、すると「本当に開いていいか」という趣旨のダイアログが出るので、「開く」をクリックしてください。Mac だとそれでもファイルの実行権限まわりのセーフティ機能が働くのか、VSM が FORTH80.bin を開けずにすぐ終了してしまいます。ですが、ターミナルから ./VSM と入力して開くと正常稼働します。いずれの場合も、以降は OS によって実行禁止されることはありません。

***注意：使用は自己責任で！***

***CAUTION: Use at your own risk!***

## 入出力について

FORTH80 は実質、エミュレータなので、標準入出力と結びつくことで初めて実 PC に作用を及ぼすことができます。それゆえ、少々イレギュラーな標準入出力の用い方をしています。

FORTH80 はデフォルトでは標準エラー出力に出力します。

>1 PFLAG !&crarr;

と入力して値 1 をユーザ変数 PFLAG に設定すると、標準出力にも出力を行います。これは、古い FORTH 処理系におけるモニタ出力とプリンタ出力とをそのまま標準エラー出力と標準出力とに当てはめたものです。ちなみに、値は非ゼロならば 1 でなくても構いません。一方、

>0 PFLAG !&crarr;

と入力すれば、FORTH80 は即座に標準出力への出力を停止します。

入力については、デフォルトの FORTH80 は内部的に C 言語の getch 関数、またはそれ相当の仕組みで標準入力を介ささず直接的にキー入力を受け付けます。起動オプション -s を付けて起動すると、標準入力から入力を行います。

デフォルトの仕様は、FORTH80 の参考元になった古い FORTH 処理系がキー入力の検知をする内部ワード（ただし未使用）を備えているのでそれに合わせるためでもありますが、日本語入力のためでもあります。Mac 版において標準入力を用いると、全角文字をバックスペースで消去したとき半角文字の幅しか戻らないという現象が発生します。二回押して全角文字一文字分のデータが消えているならこれでも問題ないのだが、どうも一回につき全角文字一つぶんデータも消去されている。この現象は、C 言語で普通に scanf 関数 を用いて書いた単純なプログラムを Mac に最初から入っている clang でコンパイルしたものでも現れます。それゆえ、いかんともしがたく、（少なくとも私の環境で）C 言語で作った実行ファイルで日本語入力を不便なく行うには、標準入力を使わないようにしなければなりません。それゆえ、Mac 版の FORTH80 で日本語を入力する必要があるときは、なるべく -s をつけずに起動してください。

言い忘れましたが、FORTH80 を標準出力にすると、画面への文字出力がすべてダブります。デフォルトで標準エラー出力に出力するようになっているのに、それに加えて標準出力も行ったからです。実は、前のバージョンまでは、排他的に出力し分ける方式にしていて、そのためにあえて FORTH 本来の内部構造を少しいじっていました。しかし、それだと標準出力にファイルを割り当てたとき一切の出力表示が消えることに遅ればせながら気付かされました。一方で、そのようにしたとき標準出力は画面に割り当てられていないから、当然ながら文字のダブりは起こりません。さて、私は FORTH80 についてリダイレクトや簡単なパイプ処理ができることにせめてもの実用性を見出しているので、出力の表示が消えることのほうが致命的だと判断しました。一方、文字のダブりについては、そもそも標準出力を表示以外のことに使うつもりでしたので、さほど困らないのです。また、先に示したように、0 PFLAG ! と入力しさえすれば直ちに収まるものでもあります。ただ、それが 00  PPFFLLAAGG  !! と表示されるわけだけれども。

## 独自の FORTH ワード

FORTH80 にはいくつか拡張的に付け足した FORTH ワードがあります。以下に見られる「(x1 x2 ... --- y1 y2 ... )」なる記法は、そのワードがスタックから値 x1, x2, ... を pop して、値 y1, y2, ... をスタックに push することを意味します。なお、厳密な記法というわけではありませんが、値が アドレス（16ビット）を意味するときは a と表記し、16 ビットの数値のときは n，8 ビットの数値のときは c，32 ビットの数値のときは d としばしば表します。

### PFLAG ( --- a )

ユーザ変数です。「入出力について」のところで説明したように、これは標準出力に出力するかどうかを決めるためのフラグです。

### UTF-8 ( --- a )

これもユーザ変数なので、自身のアドレス a を返します。

文字コードとして UTF-8 を用いるかどうかを決めるためのフラグです。非ゼロに設定すると、FORTH80 は UTF-8 に適したやり方で文字を処理します（日本語環境の漢字でしか機能しないかもしれません）。

### ECHO ( --- a )

ユーザ変数です。入力をエコーするかどうかのフラグ。このエコーはターミナルの行うそれではなくて、FORTH80 が行うそれです。フラグの値はアドレス a に格納されています。

FORTH80 が標準入力を用いているときは、ターミナル自体が入力のエコーを行うので入力の表示がダブります。それゆえ、ECHO の値をゼロにして FORTH80 側のエコーを切る必要が出てきます。一方、キーボードからの直接入力を用いているときはエコーが必要なので、ECHO の値を非ゼロに設定します。起動オプション -s をつけて FORTH80 を起動すると、ECHO の初期値が 0 に設定されます。何もつけないとき初期値は 1 です。（ちなみに、入力に関しては起動後に方式を切り替えることはできません。）

### UVR ( --- n )

これは定数であり、ユーザ変数の初期値が書かれたメモリ領域の先頭のアドレス n を返します。n と表記するのはこれが定数のためです。

FORTH80 の起動時に、ユーザ変数にそれぞれの初期値が読み込まれます。ユーザ変数は後からいくらでも値を書き換えられますが、ユーザ変数に対する通常の操作で初期値自体が書き換わることはありません。それら初期値を変更するためにこの定数 UVR が必要になります。

### SCR/DRIVE ( --- n )

仮想フロッピードライブあたりのスクリーンの数を意味する定数です。

スクリーンとは、古い FORTH 処理系における外部記憶装置のデータ管理で用いられる単位です。スクリーンあたり 1024 バイトです。

### POPENR, POPENW ( a --- )

これらは C 言語における popen 関数を模したワードです。実際にそれを使って実装してあります。実はどちらも popen の "r+" モードを使っています。

（ただし、Windows 版ではやむをえずモード "r", "w" を使っています。これは MinGW の gcc がどうやら popen の "r+" モードをサポートしていないからです。それゆえ、開いたモードと異なるモードで読み書きしようとしてエラーが出ることを避けるために、以下で述べる、入力に長さゼロあるいはヌル文字で始まる文字列を設定する使い方はしないでください。）

内部的には POPENR, POPENW それぞれ専用のファイルポインタを用いています。ゆえに、ファイルポインタの選択の自由がないところが C 言語における popen とは異なるところです。

アドレス a から始まる文字列をターミナルやコンソールのコマンドとみなして実行し、その実行、すなわちサブプロセスに対して POPENR は出力の読み出しを、POPENW は入力の書き込みを行います。前者の場合、読み出されたものがそのまま FORTH80 の標準入力になります。後者の場合、FORTH80 の標準出力がそのまま書き込みになります。

ただし、アドレス a から始まる文字列はいわゆる Pascal 文字列です。すなわち、先頭 1 バイトには文字列の長さが格納されています。

**特別な場合として、長さがゼロの文字列、もしくはヌル文字を先頭に持つ文字列が与えられた場合、POPENR と POPENW は互いに相手のすでに開いているサブプロセスの入出力を使おうとします。**

忘れてはいけないのは、**POPENR, POPENW を実行した後は必ずワード PCLOSER, PCLOSEW を実行してサブプロセスを閉じる必要がある**ことです。これは、C 言語において popen した後は pclose することと対応しています。

他にも注意すべきことがあります。例えば、POPENR を使って引数なしの cat コマンドを実行すると、一見したところキー入力はできるのですが実際は無反応です。これは引数なしで実行すると cat が入力待ちになるからですが、しかし POPENR によって cat の出力が FORTH80 の入力へと繋がれたので、FORTH80 はキー入力を受け付けることができず、一方、cat はそんな FORTH80 の出力を待っている、ということになります。この状態から FORTH80 を強制終了させずに復帰するには、どうやらそれなりに面倒くさい手段が一つしかありません。使い方によってはこのようなことになるので注意してください。

ついでだからその方法についても述べておきます。これはむしろ自分のためです。私の環境は今のところ Mac ですが、あえて Linux に寄せたターミナルを使った方法を記します。

---

まず、Control + Z キーで FORTH80 のプロセスをサスペンド。するとコマンドプロンプトが現れるので、

    % jobs -l

でサスペンドされているジョブが詳細にリスト表示される。すると確かに ./VSM と書かれたところがあって、そうなっていることが分かる。リストの左端の番号 [n] がジョブ番号。

次に、

    % ps | grep cat

と入力・実行すると、cat を含んだ名前を持つプロセスの一覧が表示される。ズバリ cat と書かれているものは大抵一つしかないからそれが目的のプロセス。

さて、プロセス名の左端に書かれたプロセスIDの番号 xxxxx をコピペして

    % kill xxxxx

と入力・実行してプロセスを停止。

最後に、ジョブ番号 n を使ってくだんのジョブをフォアグラウンドで再開させる。

    % fg %n

% 記号は省略可。

---

さて、このように、POPENR はサブプロセスの標準出力を FORTH80 の入力に戻します。それゆえ、POPENR だけを実行すると、ターミナルやコンソールに出力されるべきものがそのまま FORTH80 に流れ込んできますが、それらはもちろん FORTH80 のプログラムではないので FORTH80 はエラーを出すばかりです。いや、むしろそれだけならいいのですが、もしたまたま何かの FORTH ワードと同じスペルの言葉が含まれていたら、意図しない実行結果をもたらす恐れがあります。それゆえ、「読んで表示する」あるいは「読み捨てる」などの作業が必要です。

前者を行うならば、例えば、FORTH80 にて二つの FORTH ワードを次のようにユーザ定義します。

    : PAD<  QUERY 0 WORD PAD OVER C@ 1+ CMOVE ;

    : OUT>  PAD POPENR BEGIN KEY DUP 255 < WHILE EMIT REPEAT DROP PCLOSER ;

"255" は文字コード 0xFF を意味しています。

後者の場合は、OUT> の代わりに THR> を用意します。

    : THR>  PAD POPENR BEGIN KEY DUP 255 < WHILE DROP REPEAT DROP PCLOSER ;

ワード名は重複しさえしなければ何でもよく、一例に過ぎません。さて、このように定義した上で、PAD< と入力してリターンキーを押すと「ok」の表示が出ずにカーソルがワードのすぐ後ろ（オプション -s をつけて起動している場合は次の行の先頭）で止まって更なる入力待ちになります。そこで、コマンドを表す文字列を入力します。つまり、

>PAD<&crarr; \<command\>&crarr;

で、文字列 \<command\>（スペースを含んでもよい）が FORTH80 システム内のメモリの PAD と言われる場所に格納されます。ただし、PAD の仕様上、80 文字までしか推奨されません。メモリ領域を別途に確保すればもっと長い文字列もイレギュラーに破壊される恐れなく扱えますが、Pascal 文字列の制約上、最大でも 255 文字であることに注意してください。

たとえば、

>PAD<&crarr; ls -l&crarr;

などとします（Mac/Linux の場合。Windows では "dir"）。

その上で、

>OUT>&crarr;

とすれば、カレントディレクトリの内容が、FORTH80 であるはずの画面にずらずらと表示されるはずです。

### PCLOSER, PCLOSEW ( --- )
PCLOSER は、POPNER で開いたプロセスを閉じます。PCLOSEW は、POPENW で開いたプロセスを閉じます。ただし、**POPENR と POPENW が同じプロセスを開いているとき、先に閉じた方は実際は閉じずに単にそのプロセスの使用をやめるだけ**です。この場合は、最後にクローズされたときに真にクローズされます。

## 起動オプション

ECHO の項ですでに触れましたが、FORTH80 には起動オプションがあります（正確には VSM もしくは VSM.exe のそれ）。-h と -s の二つしかありません。

 -h は起動オプション自体のヘルプを表示します。

 -s はすでに触れましたが、標準入力を介して入力を行うかどうかです。


## 定義しておくと便利なワード

 作者自身の備忘録も兼ねて、定義しておくと便利な FORTH ワードを挙げておきます。いずれも、ワード名は単なる一例です。ちなみに、スペースを伴った左丸括弧「(」は実は FORTH ワードで、これが FORTH におけるコメント文の開始記号です。記号「)」（こちらは FORTH ワードではない）までを読み飛ばします。

### {, }
    : { ( --- ) 1 PFLAG ! ;

    : } ( --- ) 0 PFLAG ! ;

このように定義すると、ワード「{」とワード「}」に挟まれた FORTH プログラムは出力先が標準出力になります。

### .\

    : .\ ( --- ) S->D 0 D.R ;

このワード「.\」は、ワード「.」と同等の機能で、つまり、値を数字表現として（そのとき FORTH80 で設定してある進数に応じて）表示するものです。ただ、後ろにスペースが出力されないかどうかのみが異なります。「.\」はスペースなしです。

そもそも、初めから「.」をスペースなしに設定しておいたほうがいろいろと融通が効いたのではないか、と思うのですが、なぜか FORTH のデファクトスターンダード的に「.」はスペース付き出力になっている。

### HOME, CLS

    : HOME ( --- ) 27 EMIT ." [0;0H" ;

    : CLS  ( --- ) HOME 27 EMIT ." [2J" ;

HOME はカーソルを画面の左上端に移動させ、CLS はそうしたうえで画面を消去するワードです。

ただし、「ok」の表示が出るのでカーソルは完全に一番上の行に来るのではありません。

これらは要するに、ANSI エスケープシーケンスのカーソル位置指定コマンドと画面消去コマンドを使っているだけです。ところが、１行あたりの文字数が固定されていてスクロールも自由にできない昔の PC とそうではない現代の PC とでは「画面」の意味合いが異なってしまっているようです。つまり、カーソルの戻る範囲および画面の消せる範囲が限定されています。

さらに、Windows では、コンソールが ANSI エスケープシーケンスを受け付けないので、レジストリをいじるなどして設定を変えないといけないようです。
